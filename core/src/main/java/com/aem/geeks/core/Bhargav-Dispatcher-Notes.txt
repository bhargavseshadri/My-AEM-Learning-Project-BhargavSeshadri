*) DISPATCHER:
    - The AEM Dispatcher is a caching and load-balancing tool that sits between your web server (Apache/IIS) and AEM publish instances.

- Caching — Stores rendered HTML pages to reduce load on AEM
- Load Balancing — Distributes requests across multiple publish instances
- Security — Filters malicious requests and protects AEM
- URL Rewriting — Transforms URLs

=============================================================================================
*) Important points

- Dispatcher does not perform tasks, dispatcher is a module in our apache web server that tells Apache webserver what tasks need to perform.
- Caching, load balancing, access and security all these are the features of APACHE webserver, but the dispatcher tell the server
  to create the cache as per the provided configurations.
- Using rewrite rules we will redirect the pages :: Eg: if we hit www.intel.com  -> it will redirects to -> https://www.intel.com/content/www/us/en/homepage.html

*) Difference between cache deletion, cache invalidation, cache purge

- Cache Invalidation : marking cached content as stale or unusable without deleting file immediately.
                        - It works via .stat files — when content is activated, Dispatcher updates the timestamp on the relevant .stat file(s) up to the configured
                          /statfileslevel. Any cached file whose own timestamp is older than its nearest .stat file is considered stale.
                        - dispatcher fetches fresh content and overwrites cache
                        - Old file exists physically but is logically invalid.
                        - The stale file physically still sits on disk — it just gets ignored and re-fetched from AEM the next time someone requests that content.

- Cache Deletion : The literal, physical removal of files from the filesystem under the Dispatcher's /docroot.
                   so the next request goes to publisher and then page served and a new cache file got created.

- Cache Purge : “Cache purge” is most commonly associated with CDNs.
                 - it usually means Explicit forced removal of cached content from caching layer like cdn.

- cache flush : This is the umbrella term for the whole event/request that triggers invalidation (or deletion) — the mechanism that fires when a page
                is activated. On author, a replication agent sends the activated content to Publish; Publish then has its own agent (the "Dispatcher
                Flush Agent") that fires a request to Dispatcher telling it something changed. Adobe's docs use "flush" and "invalidate" almost interchangeably
                in this context — a "flush request" is what causes invalidation to happen.

2. /clientheaders :
- use of this is Forwarding HTTP Headers. it tells what headers are allowed through request.
- Controls which HTTP headers are passed from client to AEM. Dispatcher strips unnecessary headers by default for security and performance.
- when a User make a page request, that request have page URL and lot of headers, and AEM don't need all those headers, so our dispatcher only allows those headers
  we are specified under /clientheaders.

Eg:1. * Allowing every header
        /clientheaders
          {
          "*"  # Forward all headers
          }

      * Allowing only specific headers
        /clientheaders
          {
          "Host"
          "User-Agent"
          "Accept"
          "Accept-Language"
          "Authorization"
          "Referer"
          }

- For example Suppose users log in using a JWT token. If the clientheader allows "Authorization" the the dispatcher allow the JWT token and AEM checks it and allows
  users to login.
  If at all we don't allowed the "Authorization", then in that case dispatcher restricts the JWT TOKEN and AEM dont gets that token to check, in return
  AEM give 401-unauthorised.

3) /virtualhosts  — Domain Mapping, When a request reaches Dispatcher, it compares the request's Host header (and optionally the URL path)
                    with the configured /virtualhosts entries. The first matching farm is selected, and that farm's cache, filters, renders,
                    and other settings are used to process the request.
- it define which domain is mapped or resolved to which content. using this we can have multiple domains with multiple configuration.
- we can have multiple /virtualhosts.

- Eg: /farms {
          /mainfarm {
              /virtualhosts {
                  "www.company.com"
              }
          }
          /shopfarm {
              /virtualhosts {
                  "shop.company.com"
              }
          }
          /careerfarm {
              /virtualhosts {
                  "careers.company.com"
              }
          }
      }

5) /filter : when we want to filter the request we use this. basically we don't want to send all the requests to our AEM server, we only send some specific
            filtered requests. those specifications will be done here.
- The /filter section is your security gate.
- It blocks access to sensitive paths like:
              /system/console
              /crx/de
              /bin/querybuilder.json
              /etc/packages/*.zip

- Dispatcher filter rules are evaluated in order and the last matching rule wins, not the first. meaning if a request matches a deny rule
  but then also matches an allow rule further down the list, the request will be allowed despite having matched a deny rule.
- Dispatcher uses a default-deny model only explicitly allowed paths are permitted. If a URL does not explicitly match an allow
  rule, it is effectively denied. This default-deny model is intentional. You explicitly allow only what is safe.

- Basic example : /filter {
                    /0001 { /type "deny"  /url "/crx/*" }
                    /0002 { /type "deny"  /url "/system/console/*" }
                    /0003 { /type "allow" /url "/content/*" }
                    /0004 { /type "allow" /url "/etc.clientlibs/*" }
                    /0005 { /type "allow" /url "/content/dam/*" }
                    /0006 { /type "allow" /url "/dispatcher/invalidate.cache" }
                    /0007 { /type "allow" /url "/libs/granite/core/content/login.html" }
                  }



-Rule format: /XXXX { /type "allow|deny" /glob "PATTERN" }

- Default deny all: /0001 { /type "deny" /glob "*" }
- Allow specific paths
              /0010 { /type "allow" /glob "GET /content/*" }
              /0011 { /type "allow" /glob "GET /etc.clientlibs/*" }

- Block dangerous selectors
              /0050 { /type "deny" /glob "* *.infinity.json*" }
              /0051 { /type "deny" /glob "* *.tidy.json*" }
              /0052 { /type "deny" /glob "* *.sysview.xml*" }
              /0053 { /type "deny" /glob "* *.docview.*" }
              /0054 { /type "deny" /glob "* *.*[0-9].json*" }

- Method restriction:  We can also restrict by method also, in the below example we are restricting "GET" method and allowing only "POST" method.
                        /filter {
                          /0050 { /type "allow" /url "/bin/my-servlet" /method "POST" }
                        }

- Allowing a JSON Endpoint Safely : Suppose you have a component that fetches JSON data from "/content/mysite/en/home.model.json". If you get
                                    403, then we have to explicitly allow .model.json requests.
                                    /filter {
                                      /0060 { /type "allow" /url "*.model.json" }
                                    }

6) /cache : we use this for Caching configuration, The cache section regulates what responses will be cached and where.

- This section defines what to cache, where to store it, and how to invalidate it

Eg : /cache {
       /docroot "/opt/dispatcher/cache"
       /statfileslevel "2"
       /allowAuthorized "0"
       /serveStaleOnError "0"
       /rules {
         /0000 { /glob "*.html" /type "allow" }
         /0001 { /glob "*.user.json" /type "deny" }
       }

       /invalidate {
         /0000 { /glob "*" /type "allow" }
       }
     }

- Key elements
    -> /docroot – location for cached files
    -> /statfileslevel "2" – /statfileslevel controls how deep Dispatcher creates .stat files in the cache directory structure. These .stat
                             files are used for cache invalidation. When content is activated, Dispatcher updates the appropriate .stat file,
                             and cached pages whose timestamps are older than that .stat file are treated as stale and refreshed from AEM on
                             the next request.
                           - A .stat file is a small timestamp file. It does not store page content. It only stores the last modification time.
                             EG: /cache
                                 └── content
                                     └── site
                                         └── en
                                             ├── home.html
                                             ├── products.html
                                             └── about.html
                                 - /statfileslevel "2"
                                 - Result is : /cache
                                               ├── .stat          (Level 0)
                                               └── content
                                                   ├── .stat      (Level 1)
                                                   └── site
                                                       ├── .stat  (Level 2)
                                                       └── en
                                                           ├── home.html
                                                           ├── products.html
                                                           └── about.html
                                - Imagine you publish a change to: /content/site/en/home, Dispatcher updates the nearest .stat file (based on the configured level).
                                - So /content/site/.stat gets a newer timestamp, all cached pages under that part of the cache are considered potentially stale

    -> /allowAuthorized "0" : - (FLAG) this flag indicating whether to cache responses to requests that contain authorization information.
                              - here "0 means No" and "1 means Yes".

    -> /serveStaleOnError "0" : - (FLAG) Flag indicating whether the dispatcher should serve stale content if no remote server is available.
    -> /rules – which files to cache or skip
              - The rules section defines what responses should be cached based on the requested URL. Please note that only the following requests can
                lead to cacheable responses.
                         - HTTP method is GET
                         - URL has an extension
                         - Request has no query string
                         - Request has no "Authorization" header (unless allowAuthorized is 1)

7) /invalidate
- The invalidate section defines the pages that are "invalidated" after any activation.
- Please note that the activated page itself and all related documents are flushed on an modification.
  For example: if the page /foo/bar is activated, all /foo/bar.* files are removed from the cache.

EG: /invalidate
            {
            /0000
              {
              # Deny all files for invalidation
              /glob "*"
              /type "deny"
              }
            /0001
              {
              # Consider all HTML files stale after an activation.
              /glob "*.html"
              /type "allow"
              }
            /0002
              {
              /glob "/etc/segmentation.segment.js"
              /type "allow"
              }
            /0003
              {
              /glob "*/analytics.sitecatalyst.js"
              /type "allow"
              }
            }

8) /allowedClients
- This tell to the server which clients(IP addresses or hostnames) are allowed to invalidate (flush) the Dispatcher cache.
- to put it precisely, The allowedClients section restricts the client IP addresses that are not allowed to issue activation requests.

Eg:      /allowedClients
            {
              /0000
               {
                /glob "*"
                /type "deny"
                }
              /0001
                {
                /glob "127.0.0.1"
                /type "allow"
                }
             }

- What it does :- When AEM publishes content, it can send cache invalidation (flush) requests to the Dispatcher.

   * Now The /allowedClients section determines: Which IPs are allowed to send cache invalidation requests.
                                                 Which IPs are denied from clearing the Dispatcher cache.

   * If a request comes from an IP that isn't allowed, the Dispatcher ignores the invalidation request.
   * Usually only the AEM Publish instance(s) or the AEM Flush Agent IPs are allowed.

9) /ignoreUrlParams
- The ignoreUrlParams section contains query parameter names that should be ignored when determining whether some request's output can be cached
  or not.
- EG : In this example configuration, the "q" parameter will be ignored.
              /ignoreUrlParams
                {
                /0001 { /glob "*" /type "deny" }
                /0002 { /glob "q" /type "allow" }
                }

* Here - /type "allow" → Ignore the matching query parameters when caching.
       - /type "deny" → Do not ignore the matching query parameters; they influence cache behavior.

10) /headers
- This section is used to specify which HTTP response headers should be cached along with the content and returned to clients.
- By default, Dispatcher caches the response body (HTML, JSON, images, etc.), but it does not automatically cache all response
  headers. You explicitly list the headers you want it to store.
Eg : /cache
     {
         /headers
         {
             "Cache-Control"
             "Content-Type"
             "Last-Modified"
             "Expires"
             "ETag"
         }
     }

- /headers : > Caches response headers from AEM and sends them to clients
             > This comes when AEM sends response to the browser.

- /clientheaders : > Forwards request headers from the client to AEM
                   > This comes when browser Requesting AEM.

11) /gracePeriod "2"
- The /gracePeriod setting in the AEM Dispatcher dispatcher.any file specifies how long Dispatcher waits before performing another cache
  invalidation for the same content.
- Its main purpose is to reduce the impact of many cache flush requests arriving in a short period, which can happen during bulk content activation.
- The value is in seconds.
- It only affects cache invalidation (flush) behavior.
- It is especially useful when authors publish many pages or assets in a short time.

- The /gracePeriod property defines the number of seconds a stale.

Eg: /cache
    {
        /gracePeriod "2"
    }
    - After processing an invalidation, Dispatcher waits 2 seconds before processing another invalidation for the same cached content.

12) /enableTTL "1"
- The /enableTTL setting enables time-based cache expiration in AEM Dispatcher by listening the HTTP expiration headers returned by AEM.
- When enabled, Dispatcher uses headers such as: "Cache-Control: max-age=..., Expires: ..." to determine when a cached file should expire.

- How it works: * Suppose AEM responds with: HTTP/1.1 200 OK
                                           Cache-Control: max-age=300
                * This means: The page is valid for 300 seconds (5 minutes).

14) /auth_checker : Authorization checker
- before a page in the cache is delivered, a HEAD request is sent to the URL specified in /url with the query string
  '?uri='. If the response status is 200 (OK), the page is returned from the cache. Otherwise, the request is forwarded to the render and
  its response returned.
- Only pages matching the /filter section below are checked, all other pages get delivered unchecked.
-  All header lines returned from the auth_checker's HEAD request that match the /headers section will be returned as well.

EG:  /auth_checker
           {
           /url "/bin/permissioncheck.html"
           /filter
             {
             /0000
               {
               /glob "*"
               /type "deny"
               }
             /0001
               {
               /glob "*.html"
               /type "allow"
               }
             }
           /headers
             {
             /0000
               {
               /glob "*"
               /type "deny"
               }
             /0001
               {
               /glob "Set-Cookie:*"
               /type "allow"
               }
             }

  1. Auto Invalidate: When a page is activated, the cache entry is auto-expired. invalidation is generally the recommended approach.
  2. Flush Agents: Explicitly trigger cache clear.
  3. Time-based Expiry: Set TTL for cache entries.

*) In Depth on Caching and Invalidation

- To maintain fresh content, outdated cache entries must be cleared. AEM handles this through two mechanisms: a Flush Agent
  or a direct HTTP invalidation request. Both methods trigger an update to the Dispatcher’s statfiles, which is the core of AEM’s cache freshness logic.
- A statfile is essentially a timestamp. When content is published or explicitly invalidated, Dispatcher updates this timestamp.
- On every incoming request, Dispatcher performs a quick comparison:
           - If the cached file’s timestamp is older than the statfile’s timestamp, the file is treated as stale.
           - The request is then forwarded to AEM Publish, which returns fresh content that Dispatcher writes back into cache.

- Does Invalidation Clear Everything on Every Request?
Ans: No. That’s where statfileslevel matters. statfileslevel defines how deep the statfile hierarchy goes within your
     content structure. This controls the scope of invalidation.
          - A low level (e.g., 0) invalidates the entire site for every update—causing unnecessary load and cache stampedes.
          - A reasonable level (usually 2 or 3) limits invalidation to only the relevant content subtree.

---------------------------------------------------
*) Dispatcher Configuration for DYNAMIC CONTENT

1. Pages and Selectors
        - Standard HTML pages should always be cacheable.
        - When you introduce selectors, you’re effectively creating unique variations of the same resource.
        - For example, /content/site/en.light.json and /content/site/en.dark.json are two distinct representations of the
          same path. Dispatcher treats each as a completely separate cache entry.
2. Query Parameters (The Cache Killer):  The next major problem is query parameters. By nature, query parameters destroy caching because they imply
                                         that the response can change for every different parameter. Dispatcher will not cache them unless you explicitly
                                         configure rules that define which parameters matter and which should be ignored.
3. Assets and ClientLibs (Cache Forever): Assets and ClientLibs follow a different rule: anything versioned should be cached as long as possible.
                                          Versioned URLs (for example, /etc.clientlibs/site/clientlib.css?12345) inherently guarantee that the content
                                          changes only when the URL changes.

*) CDN
- Definition : CDN is a globally distributed network of edge servers that store cached versions of your site’s content like HTML, images, PDFs, JS, CSS,
               closer to users around the world. Instead of every request traveling all the way to your cloud region, the CDN responds from the nearest
               physical location.
- Dispatcher lives in one region. A CDN has hundreds of servers across continents. Users get content from the closest server, not from datacenter.

*) Dispatcher Re-Write Rules

- In AEM Dispatcher, rewrite rules are typically not written in Dispatcher itself. They are usually handled by the web server
  in front of Dispatcher.URL rewriting happens before the request reaches Dispatcher.
- Rewrite rules are usually placed in your VirtualHost configuration or an included rewrite file.
- In a standard AEM project (including AEM as a Cloud Service), these rules are typically written in your Dispatcher module,
  often inside a file like conf.d/rewrites/rewrite.rules.
- You must ensure the rewrite engine is turned on "RewriteEngine On"

1. https://www.example.com/about Internally rewrite to: /content/my-site/us/en/about.html
    Rule : RewriteEngine On
           RewriteRule ^/(.*)$ /content/my-site/us/en/$1.html [PT,L]
           - The PT (Pass Through) flag ensures the rewritten request is passed to Dispatcher.
           - [L]: Last rule. Stop processing further rewrite rules if this one matches.

2. Global root mapping: Redirecting the root domain [www.yoursite.com/](https://www.yoursite.com/) to a specific language homepage.
        - RewriteRule ^/$ /us/en.html [R=301,L]
        - [R=301]: Issues a permanent HTTP 301 redirect to the browser.

3. Redirect old URL to new URL -> RewriteRule ^/old-page/?$ /new-page [R=301,L]

*) TTL - time to live

- TTL stands for Time To Live. In the context of web performance and AEM, it refers to the amount of time a cached file (like an HTML
  page, image, or JSON response) is considered "fresh" or valid.
- When a user requests a file, the cache (like the AEM Dispatcher, a CDN, or the user's browser) checks the file's TTL.
- If the TTL has not expired, The cache serves the file directly to the user (a "cache hit"), which is extremely fast. If the TTL has expired,
  The cache considers the file "stale" and makes a trip back to the origin server (the AEM Publisher) to fetch a fresh copy, which it then
  re-caches with a new TTL.
- TTL is almost always defined in seconds using the HTTP "Cache-Control: max-age=..." header.

Setting TTL:
> Step 1: Enable TTL in the Dispatcher Farm, By default, the Dispatcher ignores time-based headers and relies on AEM publish events. You must
  explicitly configure your Dispatcher farm file.
  - Locate your /cache section and add /enableTTL "1" : /cache {
                                                            # Other cache configurations...

                                                            # Enable TTL evaluation
                                                            /enableTTL "1"
                                                        }
  - When this is set to "1", the Dispatcher will look for Cache-Control: max-age or Expires headers coming from AEM. When the max-age is reached,
    the Dispatcher will automatically delete the cached file.

> Step 2: Set the Cache-Control Headers
        Method A: Setting TTL via Apache (Best for Static Assets) For static assets that rarely change—like fonts, images, or clientlibs (CSS/JS)
                  —it is best practice to set the TTL directly in your Apache virtual host configuration using mod_headers.

        Method B: Setting TTL via AEM (Best for Dynamic Pages & API Responses) For actual AEM pages or JSON API endpoints, the TTL often needs to
                  be determined dynamically by the backend (e.g., caching a news feed for 5 minutes, but a static "About Us" page for 12 hours).
