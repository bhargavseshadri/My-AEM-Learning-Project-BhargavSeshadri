==> SERVLETS
- A Servlet is a Java class that  handles HTTP requests (like GET, POST) and generates HTTP responses (like HTML, JSON, XML).
- Servlets are of two types Resource Type Servlets and Path Type Servlets. Both types of servlets must extend either SlingAllMethodsServlet or SlingSafeMethodsServlet.

-> When a servlet runs, Sling (and the underlying web server Jetty) automatically gives it two key objects:
   SlingHttpServletRequest → the HTTP request (what the browser sent)
   SlingHttpServletResponse → the HTTP response (what you send back)

a) ResourceType Servlets
-> ResourceType, Methods, extensions and Selectors will only work for resourceType servlets. (ResourceType is manditory in these and others are optional)
-> so we give a Resource in the servlet, and whenever that resource is accessed, our servlet will be "CALLED".
-> ResourceType Servlets : @SlingServletResourceTypes()
- these servlets can be accessed controlled using a jcr repo ACL
- if we do not provide any method then the default method is GET
-> Cases when we prefer ResourceType
a) Dynamic component behavior: When the servlet needs to render or process requests specific to a component.
b) Component-based JSON export: For example, exporting data only for a news-page component.
c) Great for component-specific functionality.

b) PathType Servlets
-> Path @SlingServletPaths(value=__)
-> ACL access is not present
-> fixed url endpoint

-> Benifits of pathType
- when we want our servlet to be Independent of page/component structure then we use this pathType.
- Creating utility endpoints (like APIs)
- Handling AJAX calls from frontend
- Exposing data externally

*) SERVLETS - GET and POST requests explanation

Simple understanding
* Post : A POST servlet is used when the client sends data to the server.
            - Save data into JCR
            - Update existing content
            - Trigger some backend processing
            - Submit forms
            - Or in some places it simple send the data to the server and that data we use in our code.

* Get : A GET servlet is used when the client requests data from the server. used for mainly Reading data from JCR.

--> GET FLOW
a) Browser issues GET /path?query=abc HTTP/1.1 (headers, cookies).
b) Server/router receives request, Sling/Servlet container resolves resource/servlet.
c) Container calls doGet(req, res)
d) doGet reads req.getParameter(...), session, headers, etc., prepares content and sets response headers (Content-Type, caching),
writes body via res.getWriter() or stream.
e) Server sends HTTP status (200 OK) + body back.
f) Browser renders or caches per response headers.

--> POST FLOW
a) Browser issues POST /path HTTP/1.1 with body (form-encoded, JSON, multipart) + headers.
b) Transports to the server.
c) Container resolves servlet and calls doPost(req, res).
d) doPost reads req.getParameter() (for form), validates input, performs DB/JCR writes.
f) doPost typically returns: HttpStatus codes and then Client receives response and acts accordingly.

Q) Why in aem servlet code they use protedted void for get and post methods, what's the reason?
Ans. 1) Servlets extend a base class (SlingAllMethodsServlet or HttpServlet), In those parent classes, the doGet() and doPost() methods are already
declared as protected. So when you override them, you must keep the same access modifier (protected), otherwise Java will throw an error (you can’t reduce visibility of an overridden method).
2) they are marked as protected to limit access (i.e., only subclasses or the framework should call them).

SCHEDULERS

Defination: - A Scheduler in AEM is a background job (a Java class) that runs automatically at a fixed time or interval, without any user interaction.
            - It’s used to perform repeated or time-based tasks — like clearing cache, syncing data, sending emails, at scheduled times.

* Schedulers are of two types
1. Runnable : imp method that starts the execution is -- run()
              - implements runnable interface
              - With this we can execute only one job.

-> Job : imp method that starts the execution is -- execute(JobContext jobContext)
              - Implements JOB interface
              - With this, one scheduler can execute the job in different time frames.
              ---> USE OF THIS JOB SCHEDULER
              - Lets say we have a component and a scheduler, the component displays the data and the scheduler fetches the data from some
              service and update the data in the component. lets say we have three countries INDIA(12:00AM), GERMANY(12:00 AM), USA(12:00 AM),
              and from these three countries we are getting the data. our scheduler executes at midnight 12:00 AM and fetches the data of
              covid 19 patients from some service. Our servers are in india so for india it is fine but for the usa and germany 12AM IST
              is not a 12AM for them (INDIA 12AM -- USA 1:30PM -- GERMANY 4:30PM), so based on this to meet the requirement our scheduler should
              execute at three different times, and for that writing three Runnable Schedulers(different corns) is not a feasable way.
              In this scenario JOB SCHEDULERS enter, we can write three jobs for these three countries in one scheduler.


-> ScheduleOptions scheduleOptions = scheduler.EXPR(config.cronExpression());

*) RUN MODES

- Run Modes are used to load different configurations for different AEM environments, allowing the same application code to behave
  differently based on where it is deployed.
- Run Modes in AEM are used to identify the environment in which an AEM instance is running, such as Author, Publish, Development, or Production.
- Instead of maintaining separate codebases for author, publish, development, staging, and production, you use run modes to with different configurations.

- PURPOSE of runmodes:
        *Separate configurations for Author and Publish.
        *Maintain different settings for Dev / QA / Prod.

- When we want to identify our AEM instance uniquely in code --> (slingSettingService.getRunModes()).

- Run modes are two types.
a) Installation Run modes (Fixed Run Modes) : these we will set while installing aem instance. and we cannot change these runmodes,  these are four types
                                                1. Author
                                                2. Publish
                                                3. SampleContent  - Means aem will give you the basic sample pages like WE-RETAIL
                                                4. NoSampleContent - no sample or demo sites (means no we retail or wknd)
   - Author & Publisher, sampleContent & NoSampleContent are mutually exclusive means we cannot use them together.
   - By default if we do not set anything, aem will automatically create Author with SampleContent.

b) custom run modes : Runmodes we create like Prod, Stg, Qa, Ciqa

                                     └── config
                                     └── config.author
                                     └── config.publish
                                     └── config.dev
                                     └── config.prod

Context-Aware Configuration (CaConfig) (Have to learn in depth)
- Context-Aware Configuration (CaConfig) in AEM is a mechanism for storing and retrieving configuration values based on the content hierarchy rather
  than the AEM environment.
- Unlike OSGi configurations, which are global to an AEM instance, Context-Aware Configurations allow different sites, brands, or sections of a site to have
  different configuration values while running on the same AEM instance.
- For example, if Site A and Site B are hosted on the same AEM production environment, Site A can have support@sitea.com and Site B can have support@siteb.com
  configured separately without requiring different run modes or deployments.
- To implement it, we define a configuration interface using @Configuration, store the values under /conf/<site>/sling:configs, and access them in Sling Models
  using ConfigurationBuilder or @ContextAwareConfiguration.

SLING MODELS & SERVICES

Sling Model Defination :  - Sling Models are annotation-based Java classes that act as a bridge between the JCR and the
                            presentation layer in AEM. They adapt Sling resources or requests into Java objects, inject required dependencies
                            such as JCR properties, child resources, and OSGi services, apply business logic, and expose the final data to HTL.
                            This approach keeps business logic separate from the UI.
                          - A Sling Model in AEM is a Java class used to fetch and use content from the JCR in an easy and clean way.


* A single Sling Model can be used by multiple components. It is not tied to one component unless you explicitly restrict it.
--> Reusable across many components
                    Eg: @Model(adaptables = Resource.class)
                        public class CommonModel {
                            @Inject
                            private String title;
                        }

                        HTL : data-sly-use.model="com.project.core.models.CommonModel"


* Why we give "@Model(adaptables = {Resource.class, SlingHttpServletRequest.class}" in sling model?
Ans. It is used to specify from which objects the model can be adapted.
        - When "Resource.class" is used : Use this when your model only needs resource-level data — properties, child resources, resource type —
          and doesn't care about the HTTP request at all.

        - Use this when the model needs request-specific context that a bare Resource doesn't carry:
                          Selectors, extension, suffix
                          Request parameters / query string
                          Request headers, cookies
                          @RequestAttribute injections



**Very Very Important :: Life Cycle Of Sling Model
 -> Sometimes in sling model lines like below will through error by taking null values. and they got fixed when we move those lines in side a @PostConstruct Method
    or and other method. Why Because.
        * Eg Line : Page page = pageManager.getContainingPage(resource);

 -> Classic Tip : Do all the field initializations(assigning values to the variables) in sling model inside the @PostConstruct method or inside any other method.

-> Sling Model Life Cycle Steps: Step: 1 - Object is created (constructor runs)
                                            -> Now at this moment all the @Inject fields or @ScriptVariable fields are not yet being set or injected.
                                            -> But at this very moment if there is any fields then our sling model will run initialization for those
                                               too so in this time those fields like above get Null values.

                                 Step: 2 - - After sometime the object is created, then the @Inject and @ScriptVariable fields are injected by sling (values are set).
                                           - But by this time if we wrote those line outside the method, then sling won't go back and look for those and set the values,
                                             so those lines will keep taking null values.

                                 Step: 3 - After all the fields are injected, then @PostConstruct method runs, so if we want to use those fields
                                           then we have to use them inside this method or any other method which is called after this method.
                                         - So by this time all the injections were done rt, so the field will get the appropriate value.

                                 Step: 4 - Finally, the model is ready to be used in HTL or backend code.

- Editable templates are the modern, flexible templates introduced in AEM 6.2+. They are stored under /conf, and allow authors (not just developers) to

- Controlled via Content Policies in Template Editor.
- no developer needed to change structure
- These templates are Dynamic (author controlled)

WORKFLOWS in AEM

- workflows allow us to automate a task or a series of tasks. these tasks are most performed on content like pages or assets.
- A workflow needs a workflow model, and the WF model contains a series of steps.
- Workflow model usually stored in two places in our jcr
        1. Design Model - Stored Under : /conf/global/settings/workflow/models
        2. Runtime Model - Stored Under : /var/workflow/models   -> when the workflow executes then this run time model will be used.
- Two make the changes go from Design to Runtime we have to click the "SYNC".

--> Basically to trigger a workflow we need Two things 1.Workflow Model, 2.Payload(means the content path on which we want to run our WF, like page, asset etc)
        - Incase of triggering WF using code, then in that case to get this Payload we will use "WorkflowSession"



- Workflow Steps :
        1. Container Step : When we want to trigger an another model or another WF, in those case we use this step
        2. Participant step : When we want a human interraction in our WF, like assigning a task to a human, in those case we use this step.
        3. Decision Step : Performing a task based on a condition, mainly when we need a branching in our WF, in those case we use this step.
                                -> This is two types : OR, AND
        4. Process Step : Custom WF processess are used using this process step. we will write our own custom process code and then we
                          link that code with this step.
        5. Dynamic Participant Step : here this step will assign the participant dynamically at the time of workflow execution.

- Calling of the Model : there are mainly 3 ways
                                  1. Manual Trigger
                                  2. Using Launcher : automating the trigger
                                  3. Using API/Code : we can trigger the WF using any of the backend module like servlet, scheduler, Models etc

* DAM UPDATE ASSET WF STEPS explanation
        1. Continue updating? step -- Decision step : This process checks if the complete update process has to be executed. The workflow should terminate
        2. Metadata Processor step -- Extract metadata from the asset and store those values in asset metadata node
        3. FFmpeg Thumbnails step -- Generate thumbnails for videos.
        4. EPS Thumbnails step -- Create previews for EPS files. eps files cannot be displayed directly so this step creates a png files for thumbnails.
        5. FFmpeg Transcoding -- Create video renditions.
        6. Media Extraction -- Extract embedded content.
        7. Process Thumbnails -- Generate standard DAM thumbnails.

BUNDLE LIFECYCLE

1. INSTALLED : The bundle has been successfully uploaded and installed into the OSGi container, but its dependencies have not yet been satisfied.
2. RESOLVED : All required dependencies have been resolved. The bundle is ready to start.
3. STARTING : The bundle is in the process of being started; its BundleActivator.start() method (if any)is being called.
              Bundles using lazy activation sit in this state until a class is actually loaded that triggers activation.
4. ACTIVE : The bundle has been successfully started and is running normally. This is the healthy/desired end state for most bundles.
5. STOPPING : The bundle is in the process of stopping. Its BundleActivator.stop() method is being executed.
6. UNINSTALLED : The bundle has been completely removed from the AEM instance. This is a terminal state. The bundle can no longer transition to any other state,
                 and its resources have been released by the OSGi framework.

* FRAGMENT : A Fragment is a special type of OSGi bundle. It's not actually a lifecycle state defined by OSGi, but in AEM's /system/console/bundles,
             you'll often see a bundle labeled as Fragment because it is a fragment bundle rather than a regular bundle.
             - Cannot run independently.
             - Has no lifecycle of its own (it never becomes Active).
             - Attaches to a host bundle at resolve time
             - Simple interview ans : Fragment is a bundle type that AEM/Felix displays in the console to indicate the bundle is a fragment
                                      and therefore has no independent lifecycle. It attaches to a host bundle and cannot be started or stopped on its own.

CSRF (Cross-Site Request Forgery) :
        - In some cases a malicious site in the other tab may trick a logged-in user’s browser into making a request to another site (like AEM)
          without the user realizing it
        - When you make a POST request to an AEM servlet, you need to include a special token called the CSRF token to ensure that the request is
          legitimate and not coming from a malicious source. otherwise the request wont reach AEM.
        - So the rule AEM follows is "I will only accept POST(or PUT, DELETE) requests that include a secret token I generated".
        - Without this csrf token, AEM will reject the POST request with a 403 Forbidden error, even if the user is authenticated.

* Sometimes we include CSRF clientlib in our code, so that The client library automatically:
                         - obtains the CSRF token from AEM, inserts it into forms or AJAX requests, ensures POST requests include the expected token.
                         - Eg: including csrf clientlib - apps/aemgeeks/components/content/registrationformservlet/registrationformservlet.html
                               - We can also include that in JS as wll as a request header.

*) GRAPHQL

- In AEM, GraphQL is mainly used when your content is stored as Content Fragments and some external application needs that content.
- Usually GraphQL queries are written in the external application, not in AEM.
        -> The React or other external applications sends this query to AEM using post request. and aem takes that query reads it and sends response.

- The work of traditional api's and graphql is almost same, sending the content to the external application, then whats the advantage of GRAPHQL
        -> lets say If a React app wants only the title and author, traditional APIs may still send complete data (we can write the code to filter
           out the ones that are not reqd but that is not a advaisable standard)
                json : {
                         "title": "AEM Basics",
                         "author": "Bhargav",
                         "description": "Learning AEM GraphQL"
                       }
        -> But with graphql,
            - The React app can ask:
                Graphql: {
                           articleByPath(
                             _path: "/content/dam/articles/aem-basics"
                           ) {
                             title
                             author
                           }
                         }

            - AEM returns only:
                    {
                      "title": "AEM Basics",
                      "author": "Bhargav"
                    }
            - The consumer can decide what fields it wants.

- -> GRAPHIQL EDITOR : http://localhost:4502/aem/graphiql.html
  -> To see graphql endpoints in crxde : /content/cq:graphql/global/endpoint
  -> to create graphql endpoint : http://localhost:4502/libs/cq/graphql/sites/admin/content/console.html

- Types the external application request the data:
        1. Persisted Query
                - we write a query in our graphql editor and save that file and aem create a url for that file containing query,
                  that url is called persisted query url.
                - React or external application simply uses that url and requests the data, without writing the whole query.
                - eg: we created : http://localhost:4502/graphql/execute.json/global/bhargavFirstQuery
                      in react we simply call : GET /graphql/execute.json/global/bhargavFirstQuery
                - This is advisable because it is simple and faster.

        2. In react we write the entire query and sends it in POST request. this is not advisable because in every request we send the query
           and it make the system slower.

- You can find the example graphql queries in com/aem/geeks/core/Bhargav-notes.txt

* What happens during object creation in Java?
            When you create an object using new, the following happens:
            - JVM allocates memory for the object.
            - All instance variables are initialized to their default values (e.g., 0, null, false).
            - The constructor of the class is called.
            - If it has a parent class, the parent constructor executes first (super()).
            - Finally, a reference to the object is returned.

        - It’s stored in String Pool.
        - Helps with caching and security (e.g., for passwords, URLs).
        - Prevents data corruption when shared among multiple threads.
* What’s String Pool?
        - A special memory area in heap where String literals are stored. If two strings have same literal, both refer to the same object to save memory.
          String s1 = "Java";
          String s2 = "Java"; // refers to same object

1) Call by value : In Java, call by value means that when you pass a variable to a method, a copy of the variable’s value is passed — not the actual
variable itself. So, any changes made to the parameter inside the method do not affect the original variable outside the method.

2) Garbage Collection : Garbage Collection in Java is a process by which Java automatically removes unused or unreferenced objects
                        from memory (the heap) to free up space.

- When you create objects in Java using new, they take up memory. If no part of your program is using that object anymore (no references to it), Java will
   automatically detect it and delete it from memory — this process is called Garbage Collection.
   Eg:- Person p1 = new Person(); // object created
        p1.name = "Alice";
        p1 = null; // no reference to the object anymore, The Person object is unreachable, Garbage Collector can remove it from memory

- Types of memories in java:
    a)Stack: > For Reference variables, method calls and local variables Stack memory is used.
             > And Stack memory is also used for method execution.
             > The memory is automatically cleared when the method finishes.

    b)Heap : > For objects and instance variables.
             > Heap memory is used to store objects created using new.
             > It is shared by all parts of the program.
             > Objects in the heap stay there until they are no longer referenced, then garbage collection removes them.
             > String pool is also part of this.

3) STATIC
Example Code : com/aem/geeks/core/javarelatedcodes/StaticExamples.java
- static in Java means something that belongs to the class, not to any object, and it "can be accessed" without creating
  an object using classname.
- The main use of static is to define common things that should be shared across all objects — and to access them without creating
  an object. and it is helpful for memory management.
* To access a non-static method in static method, we have to create an object.
* Static method can only access the static variables & static methods.
* Non-Static can access both static and non-static methods.
* Static does not support "this" and "super"
* And we don't give the access modifier for the variables defined inside the static method.

5) ABSTRACT CLASS

- An abstract class in Java is a class that cannot be instantiated on its own and can have abstract methods (methods without a body)
  as well as regular methods (with a body).
- Declared using the keyword abstract.
- Can have both abstract and concrete methods.
- A subclass that extends an abstract class must implement all abstract methods (unless it’s also abstract).
- Used To define a template for subclasses.
- To allow code reuse by including concrete methods in the abstract class.
- It can have a constructor.

eg:
abstract class Animal {
    // Abstract method (no body)
    abstract void sound();

    // Regular method
    void sleep() {
        System.out.println("Animal is sleeping");
    }
}

-> Animal obj = new Animal(); //Not allowed

Note: Cannot be instantiated - means that you cannot create an object of that class directly using the new keyword.

6) INTERFACES
- Its like a plan for a classes
- Interfaces are used to define a contract that classes must follow. Any class that implements an interface must provide the implementation
  for its abstract methods.
- Interfaces contains only abstract methods.
- In interfaces all the methods are abstract and all the variables are public, static, final
- Using this we can achieve multiple inheritance (since Java does not support multiple class inheritance).
- we cannot create object for interfaces.
- It can't have a constructor.


* Example of Interface Inheritance:
1. The Base Interface (Vehicle.java)

        public interface Vehicle {
            void startEngine();
        }

2. The Extended Interface (ElectricVehicle.java)
- any class that implements ElectricVehicle is now forced to sign both contracts: it must know how to chargeBattery() AND how to startEngine().

        public interface ElectricVehicle extends Vehicle {
            void chargeBattery();
        }

3. The Implementing Class (Tesla.java)
- When a class implements the child interface, it must provide the code for all methods in the child interface and all methods in the parent interface.

        // Tesla only needs to say it implements ElectricVehicle,
        // but it is legally bound to the Vehicle contract too.
        public class Tesla implements ElectricVehicle {

            // Method from the parent interface (Vehicle)
            @Override
            public void startEngine() {
                System.out.println("Tesla engine started: (Complete silence).");
            }

            // Method from the child interface (ElectricVehicle)
            @Override
            public void chargeBattery() {
                System.out.println("Tesla is plugging into the Supercharger.");
            }
        }

4. The Main Class (Main.java)
        public class Main {
            public static void main(String[] args) {

                // We can use the child interface as the type
                ElectricVehicle myEv = new Tesla();
                myEv.startEngine();    // Works! (Inherited from Vehicle)
                myEv.chargeBattery();  // Works! (Defined in ElectricVehicle)

                System.out.println("---");

                // We can ALSO use the parent interface as the type
                Vehicle myStandardVehicle = new Tesla();
                myStandardVehicle.startEngine(); // Works!

                myStandardVehicle.chargeBattery(); // ERROR! Even though it is a Tesla, because we stored it as a 'Vehicle',
                // the program doesn't know it can charge a battery.
            }
        }

OPTIONAL OOPS (Association, Aggregation, Composition)


a) Association : (here both classes are independent) Association is a relationship between two separate classes where one class uses or
                 interacts with another class, but both can exist independently. It represents a “uses-a” relationship.

- Imp - if two classes use each other in methods
    --> void teach(Student s)
    --> Now this is usually Association.

Eg:
class Teacher {
    String name;
    Teacher(String name) { this.name = name; }
}

class Student {
    String name;
    Student(String name) { this.name = name; }

    void assignTeacher(Teacher t) {  //here we are using the Association
        System.out.println(this.name + " is assigned to teacher " + t.name);
    }
}

public class Main {
    public static void main(String[] args) {
        Teacher t1 = new Teacher("Mr. Sharma");
        Student s1 = new Student("Bhargav");

        s1.assignTeacher(t1); // Bhargav is assigned to teacher Mr. Sharma
    }
}


----------------------------------
2) Aggregation (here both classes are independent)
- Aggregation is a special form of association that represents a “Has-A” relationship between two classes.
- Aggregation is a relationship where one class owns or uses another class, but the contained object can exist independently of the container object.
  The part can exist without the whole.
- Has-A relationship but can live without it.

- Imp - if one class store a reference to another as a member variable
        --> private Teacher teacher;
        --> Then it is Aggregation

EG:
class Department {
    String name;
    Department(String name) { this.name = name; }
}

class College {
    String name;
    Department dept;    //AGGREGATION  <-------------------------

    College(String name, Department dept) {
        this.name = name;
        this.dept = dept;
    }

    void show() {
        System.out.println(name + " has department " + dept.name);
    }
}

public class Main {
    public static void main(String[] args) {
        Department d = new Department("Computer Science");
        College c = new College("SRKR College", d);
        c.show(); // SRKR College has department Computer Science
    }
}

----------------------------------
3) Composition (Both classes are dependable)
- Composition is a stronger form of aggregation. In composition, the contained object cannot exist without the container —
  both have a dependent life cycle.
- Has-A relationship and cannot live without it.

- Imp - Composition Creates its own object.

Eg:
class Engine {
    void start() {
        System.out.println("Engine started");
    }
}

class Car {
    private Engine engine;

    Car() {
        engine = new Engine(); // created inside the class → tightly bound    <----------------------------
    }

    void startCar() {
        engine.start();
        System.out.println("Car started");
    }
}

JAVASCRIPT - HOW TO GET HOLD OF DOM ELEMENTS

-> DOM means Document Object Model. In JavaScript, we use DOM methods to select HTML elements from the page and then read/change them.

-> document.getElementById("idName")
   - Selects one element by its id.
   - It returns a single element because id should be unique on the page.

   - HTML Example: <h1 id="title">Welcome</h1>
   - Example: document.getElementById("title")


-> document.querySelector("cssSelector")
   - Selects the first matching element using a CSS selector.
   - We can select by id, class, tag, attribute, etc.

   - HTML Example: <input name="./ctaText" class="cmp-title" />
   - Example: document.querySelector(".cmp-title")
   - Example: document.querySelector("[name='./ctaText']")


-> document.querySelectorAll("cssSelector")
   - Selects all matching elements using a CSS selector.
   - It returns a NodeList, so we can loop through it using forEach.

   - HTML Example: <button class="cmp-button">Save</button> <button class="cmp-button">Cancel</button>
   - Example: document.querySelectorAll(".cmp-button")


-> document.getElementsByClassName("className")
   - Selects all elements having the given class name.
   - It returns an HTMLCollection.

   - HTML Example: <div class="active">Tab 1</div>
   - Example: document.getElementsByClassName("active")


-> document.getElementsByTagName("tagName")
   - Selects all elements with the given HTML tag.

   - HTML Example: <input type="text" value="Bhargav" />
   - Example: document.getElementsByTagName("input")


-> document.getElementsByName("nameValue")
   - Selects elements by their name attribute.
   - Mostly useful for form fields.

   - HTML Example: <input name="./ctaText" value="Read More" />
   - Example: document.getElementsByName("./ctaText")


-> element.closest("cssSelector")
   - Starts from the current element and searches upward through its parents.
   - It returns the nearest parent/self that matches the selector
   r.
   - HTML Example: <coral-dialog><input name="./ctaText" /></coral-dialog>
   - Example: element.closest("coral-dialog")


-> element.parentElement
   - Gives the direct parent element of the selected element.
   - Useful when we need to move one level up in the DOM.
   - HTML Example: <div class="parent"><span id="child">Text</span></div>


-> element.children
   - Gives only the direct child elements of the selected element.
   - It does not include text nodes.
   - HTML Example: <ul id="list"><li>One</li><li>Two</li></ul>


-> element.nextElementSibling
   - Gives the next element at the same level.
   - Useful when we want to access the element immediately after the current element.
   - HTML Example: <label>Title</label><input type="text" />


-> element.previousElementSibling
   - Gives the previous element at the same level.
   - Useful when we want to access the element immediately before the current element.
   - HTML Example: <label>Title</label><input type="text" />
