==> SERVLETS
- A Servlet is a Java class that  handles HTTP requests (like GET, POST) and generates HTTP responses (like HTML, JSON, XML).

-> When a servlet runs, Sling (and the underlying web server Jetty) automatically gives it two key objects:
   SlingHttpServletRequest → the HTTP request (what the browser sent)
   SlingHttpServletResponse → the HTTP response (what you send back)

a) ResourceType Servlets
-> ResourceType, Methods, extensions and Selectors will only work for resourceType servlets. (ResourceType is manditory in these and others are optional)
-> so we give a Resource in the servlet, and whenever that resource is accessed, our servlet will be "CALLED".
-> ResourceType Servlets : @SlingServletResourceTypes()
-> Here we can use extensions, Selectors, and methods
- these servlets can be accessed controlled using a jcr repo ACL
- if we do not provide any method then the default method is GET
-> Cases when we prefer ResourceType
a) Dynamic component behavior: When the servlet needs to render or process requests specific to a component.
b) Component-based JSON export: For example, exporting data only for a news-page component.
c) Great for component-specific functionality.

b) PathType Servlets
-> Path @SlingServletPaths(value=__)
-> ACL access is not present
-> fixed url endpoint

-> Benifits of pathType
- when we want our servlet to be Independent of page/component structure then we use this pathType.
- Creating utility endpoints (like APIs)
- Handling AJAX calls from frontend
- Exposing data externally

Q) Why in aem servlet code they use protedted void for get and post methods, what's the reason?
Ans. 1) Servlets extend a base class (SlingAllMethodsServlet or HttpServlet), In those parent classes, the doGet() and doPost() methods are already
declared as protected. So when you override them, you must keep the same access modifier (protected), otherwise Java will throw an error (you can’t reduce visibility of an overridden method).
2) they are marked as protected to limit access (i.e., only subclasses or the framework should call them).

==> cardinality = ReferenceCardinality.MULTIPLE, policy = ReferencePolicy.DYNAMIC
- sometimes, we may have: Multiple implementations of the same interface (or) Services that appear or disappear dynamically (e.g., bundles starting/stopping).
, we may want to handle such changes gracefully. then in that case ReferenceCardinality and ReferencePolicy come into play.
- ReferenceCardinality → How many services can be injected
                    a) MANDATORY : Must have exactly one service available. If not present, this component won’t activate.
                    b) OPTIONAL : May or may not have one service
                    c) MULTIPLE  : Can have many implementations injected (a list).
                    d) AT_LEAST_ONE : Must have at least one available.

- ReferencePolicy → When to update the reference
                  a) STATIC :  It won't considers the unbind method with bind method. Injects dependency only once during activation.
                  b) DYNAMIC : It will automatically considers the unbind method with bind method. Automatically updates the reference at runtime
                                when services appear/disappear.

Context-Aware Configuration (CaConfig)
- Context-Aware Configuration (CaConfig) in AEM is a mechanism for storing and retrieving configuration values based on the content hierarchy rather
  than the AEM environment.
- Unlike OSGi configurations, which are global to an AEM instance, Context-Aware Configurations allow different sites, brands, or sections of a site to have
  different configuration values while running on the same AEM instance.
- For example, if Site A and Site B are hosted on the same AEM production environment, Site A can have support@sitea.com and Site B can have support@siteb.com
  configured separately without requiring different run modes or deployments.
- To implement it, we define a configuration interface using @Configuration, store the values under /conf/<site>/sling:configs, and access them in Sling Models
  using ConfigurationBuilder or @ContextAwareConfiguration.

SLING MODELS & SERVICES

Sling Model Defination :  - A Sling Model in AEM is a Java class used to fetch and use content from the JCR in an easy and clean way.
                          - It connects the AEM component (HTML + dialog data) with the Java backend, so you can access component properties
                            directly as Java variables instead of manually reading from the repository.
                          - It contains component specific logic and it always a dependent on component.

* A single Sling Model can be used by multiple components. It is not tied to one component unless you explicitly restrict it.
--> Scenario 1 — Reusable across many components
                    Eg: @Model(adaptables = Resource.class)
                        public class CommonModel {
                            @Inject
                            private String title;
                        }

                        HTL : data-sly-use.model="com.project.core.models.CommonModel"

--> Scenario 2 — Restricted to specific components
                    Eg: @Model(
                            adaptables = Resource.class,
                            resourceType = {     //so this sling model can only be used for these listed components.
                                "project/components/componentA",
                                "project/components/componentB"
                            }
                        )
                        * then the same model is usable only for those listed components.

* Why we give "@Model(adaptables = {Resource.class, SlingHttpServletRequest.class}" in sling model?
Ans. It is used to specify from which objects the model can be adapted.
        - When the model is used in HTL, it is usually adapted from the SlingHttpServletRequest
        - When the model is used in backend code (services, utilities), it may be adapted directly from the Resource
                - MyModel model = resource.adaptTo(MyModel.class); ==>  For this, the model must support Resource.class.

**Very Very Important :: Life Cycle Of Sling Model
 -> Sometimes in sling model lines like below will through error by taking null values. and they got fixed when we move those lines in side a @PostConstruct Method
    or and other method. Why Because.
        * Eg Line : Page page = pageManager.getContainingPage(resource);

 -> Classic Tip : Do all the field initializations(assigning values to the variables) in sling model inside the @PostConstruct method or inside any other method.

-> Sling Model Life Cycle Steps: Step: 1 - Object is created (constructor runs)
                                            -> Now at this moment all the @Inject fields or @ScriptVariable fields are not yet being set or injected.
                                            -> But at this very moment if there is any fields then our sling model will run initialization for those
                                               too so in this time those fields like above get Null values.

                                 Step: 2 - - After sometime the object is created, then the @Inject and @ScriptVariable fields are injected by sling (values are set).
                                           - But by this time if we wrote those line outside the method, then sling won't go back and look for those and set the values,
                                             so those lines will keep taking null values.

                                 Step: 3 - After all the fields are injected, then @PostConstruct method runs, so if we want to use those fields
                                           then we have to use them inside this method or any other method which is called after this method.
                                         - So by this time all the injections were done rt, so the field will get the appropriate value.

                                 Step: 4 - Finally, the model is ready to be used in HTL or backend code.

-> adaptables = {Resource.class, SlingHttpServletRequest.class}

WORKFLOWS in AEM

- workflows allow us to automate a task or a series of tasks. these tasks are most performed on content like pages or assets.
- A workflow needs a workflow model, and the WF model contains a series of steps.
- Workflow Steps are of Four Types:
        1. Container Step : When we want to trigger an another model or another WF, in those case we use this step
        2. Participant step : When we want a human interraction in our WF, like assigning a task to a human, in those case we use this step.
        3. Decision Step : Performing a task based on a condition, mainly when we need a branching in our WF, in those case we use this step.
                                -> This is two types : OR, AND
        4. Process Step : Custom WF processess are used using this process step. we will write our own custom process code and then we
                          link that code with this step.
        5. Dynamic Participant Step : here this step will assign the participant dynamically at the time of workflow execution.

- Calling of the Model : there are mainly 3 ways
                                  1. Manual Trigger
                                  2. Using Launcher : automating the trigger
                                  3. Using API/Code : we can trigger the WF using any of the backend module like servlet, scheduler, Models etc

CLIENTLIBRARIES

* In Adobe Experience Manager (AEM), a client library (or clientlib) is a container for client-side resources such as CSS, JavaScript, images, and fonts.
It allows you to organize and manage these resources in a modular way, making it easier to include them in your AEM components and pages.
Client libraries help optimize the delivery of front-end assets and improve website performance.
* Including it in component or editable templates/page components -->
  - Jsp : <cq:includeClientLib categories="myproject.dialog"/>
  - HTL : <sly data-sly-use.clientlib="/libs/granite/sightly/templates/clientlib.html"/>
          <sly data-sly-call="${clientlib.css @ categories='bhargav.apiStyles'}"/>

* Why we give "allowProxy = true" in AEM clientlibrary properties?
Ans. - When you create a Client Library (clientlib) under /apps/..., by default it cannot be accessed publicly to anonymous users — because the /apps path is not exposed on
       the publish instance (for security reasons). To make it accessible in a safe and public way, we set this property.
     - If you forget it On publish, clientlibs won’t load → 404 errors in browser console.
     - When allowProxy=true, AEM allows the clientlib to be accessed through a proxy path: /etc.clientlibs/<your-project>/clientlibs/<clientlib-name>

* What is the use of "Embed"(like embed="[my.site.A], ) property in client library?
Ans : What it actually do is it, if we give some client library name(here my.site.A) in the embed property, then it will automatically include that client library
      inside the current client library, so we don't have to include that client library separately in our component or page, it will automatically
      get included in this client library because of this embed property. it will simply merge all those other client lib contents in to this client lib and
      render it.

* extraClientlibs="[bhargav.dialog.validation]"
- we give this when we want our clientlibs to load at the time when we open the "comp authoring dialog".
- we give this in our comp dialog XML.

BUNDLE LIFECYCLE

1. INSTALLED : The bundle has been successfully uploaded and installed into the OSGi container, but its dependencies have not yet been satisfied.
2. RESOLVED : All required dependencies have been resolved. The bundle is ready to start.
3. STARTING : The bundle is in the process of being started; its BundleActivator.start() method (if any)is being called.
              Bundles using lazy activation sit in this state until a class is actually loaded that triggers activation.
4. ACTIVE : The bundle has been successfully started and is running normally. This is the healthy/desired end state for most bundles.
5. STOPPING : The bundle is in the process of stopping. Its BundleActivator.stop() method is being executed.
6. UNINSTALLED : The bundle has been completely removed from the AEM instance. This is a terminal state. The bundle can no longer transition to any other state,
                 and its resources have been released by the OSGi framework.

* FRAGMENT : A Fragment is a special type of OSGi bundle. It's not actually a lifecycle state defined by OSGi, but in AEM's /system/console/bundles,
             you'll often see a bundle labeled as Fragment because it is a fragment bundle rather than a regular bundle.
             - Cannot run independently.
             - Has no lifecycle of its own (it never becomes Active).
             - Attaches to a host bundle at resolve time
             - Simple interview ans : Fragment is a bundle type that AEM/Felix displays in the console to indicate the bundle is a fragment
                                      and therefore has no independent lifecycle. It attaches to a host bundle and cannot be started or stopped on its own.

6) INTERFACES
- Its like a plan for a classes
- Interfaces are used to define a contract that classes must follow. Any class that implements an interface must provide the implementation
  for its abstract methods.
- Interfaces contains only abstract methods.
- In interfaces all the methods are abstract and all the variables are public, static, final
- Using this we can achieve multiple inheritance (since Java does not support multiple class inheritance).
- we cannot create object for interfaces.
- It can't have a constructor.


* Example of Interface Inheritance:
1. The Base Interface (Vehicle.java)

        public interface Vehicle {
            void startEngine();
        }

2. The Extended Interface (ElectricVehicle.java)
- any class that implements ElectricVehicle is now forced to sign both contracts: it must know how to chargeBattery() AND how to startEngine().

        public interface ElectricVehicle extends Vehicle {
            void chargeBattery();
        }

3. The Implementing Class (Tesla.java)
- When a class implements the child interface, it must provide the code for all methods in the child interface and all methods in the parent interface.

        // Tesla only needs to say it implements ElectricVehicle,
        // but it is legally bound to the Vehicle contract too.
        public class Tesla implements ElectricVehicle {

            // Method from the parent interface (Vehicle)
            @Override
            public void startEngine() {
                System.out.println("Tesla engine started: (Complete silence).");
            }

            // Method from the child interface (ElectricVehicle)
            @Override
            public void chargeBattery() {
                System.out.println("Tesla is plugging into the Supercharger.");
            }
        }

4. The Main Class (Main.java)
        public class Main {
            public static void main(String[] args) {

                // We can use the child interface as the type
                ElectricVehicle myEv = new Tesla();
                myEv.startEngine();    // Works! (Inherited from Vehicle)
                myEv.chargeBattery();  // Works! (Defined in ElectricVehicle)

                System.out.println("---");

                // We can ALSO use the parent interface as the type
                Vehicle myStandardVehicle = new Tesla();
                myStandardVehicle.startEngine(); // Works!

                myStandardVehicle.chargeBattery(); // ERROR! Even though it is a Tesla, because we stored it as a 'Vehicle',
                // the program doesn't know it can charge a battery.
            }
        }

OPTIONAL OOPS (Association, Aggregation, Composition)


a) Association : (here both classes are independent) Association is a relationship between two separate classes where one class uses or
                 interacts with another class, but both can exist independently. It represents a “uses-a” relationship.

- Imp - if two classes use each other in methods
    --> void teach(Student s)
    --> Now this is usually Association.

Eg:
class Teacher {
    String name;
    Teacher(String name) { this.name = name; }
}

class Student {
    String name;
    Student(String name) { this.name = name; }

    void assignTeacher(Teacher t) {  //here we are using the Association
        System.out.println(this.name + " is assigned to teacher " + t.name);
    }
}

public class Main {
    public static void main(String[] args) {
        Teacher t1 = new Teacher("Mr. Sharma");
        Student s1 = new Student("Bhargav");

        s1.assignTeacher(t1); // Bhargav is assigned to teacher Mr. Sharma
    }
}

2) Aggregation (here both classes are independent)
- Aggregation is a special form of association that represents a “Has-A” relationship between two classes.
- Aggregation is a relationship where one class owns or uses another class, but the contained object can exist independently of the container object.
  The part can exist without the whole.
- Has-A relationship but can live without it.

- Imp - if one class store a reference to another as a member variable
        --> private Teacher teacher;
        --> Then it is Aggregation

EG:
class Department {
    String name;
    Department(String name) { this.name = name; }
}

class College {
    String name;
    Department dept;    //AGGREGATION  <-------------------------

    College(String name, Department dept) {
        this.name = name;
        this.dept = dept;
    }

    void show() {
        System.out.println(name + " has department " + dept.name);
    }
}

public class Main {
    public static void main(String[] args) {
        Department d = new Department("Computer Science");
        College c = new College("SRKR College", d);
        c.show(); // SRKR College has department Computer Science
    }
}

- Composition is a stronger form of aggregation. In composition, the contained object cannot exist without the container —

- Has-A relationship and cannot live without it.

JAVASCRIPT - HOW TO GET HOLD OF DOM ELEMENTS

-> DOM means Document Object Model. In JavaScript, we use DOM methods to select HTML elements from the page and then read/change them.

-> document.getElementById("idName")
   - Selects one element by its id.
   - It returns a single element because id should be unique on the page.

   - HTML Example: <h1 id="title">Welcome</h1>
   - Example: document.getElementById("title")


-> document.querySelector("cssSelector")
   - Selects the first matching element using a CSS selector.
   - We can select by id, class, tag, attribute, etc.

   - HTML Example: <input name="./ctaText" class="cmp-title" />
   - Example: document.querySelector(".cmp-title")
   - Example: document.querySelector("[name='./ctaText']")


-> document.querySelectorAll("cssSelector")
   - Selects all matching elements using a CSS selector.
   - It returns a NodeList, so we can loop through it using forEach.

   - HTML Example: <button class="cmp-button">Save</button> <button class="cmp-button">Cancel</button>
   - Example: document.querySelectorAll(".cmp-button")


-> document.getElementsByClassName("className")
   - Selects all elements having the given class name.
   - It returns an HTMLCollection.

   - HTML Example: <div class="active">Tab 1</div>
   - Example: document.getElementsByClassName("active")


-> document.getElementsByTagName("tagName")
   - Selects all elements with the given HTML tag.

   - HTML Example: <input type="text" value="Bhargav" />
   - Example: document.getElementsByTagName("input")


-> document.getElementsByName("nameValue")
   - Selects elements by their name attribute.
   - Mostly useful for form fields.

   - HTML Example: <input name="./ctaText" value="Read More" />
   - Example: document.getElementsByName("./ctaText")


-> element.closest("cssSelector")
   - Starts from the current element and searches upward through its parents.
   - It returns the nearest parent/self that matches the selector
   r.
   - HTML Example: <coral-dialog><input name="./ctaText" /></coral-dialog>
   - Example: element.closest("coral-dialog")


-> element.parentElement
   - Gives the direct parent element of the selected element.
   - Useful when we need to move one level up in the DOM.
   - HTML Example: <div class="parent"><span id="child">Text</span></div>


-> element.children
   - Gives only the direct child elements of the selected element.
   - It does not include text nodes.
   - HTML Example: <ul id="list"><li>One</li><li>Two</li></ul>


-> element.nextElementSibling
   - Gives the next element at the same level.
   - Useful when we want to access the element immediately after the current element.
   - HTML Example: <label>Title</label><input type="text" />


-> element.previousElementSibling
   - Gives the previous element at the same level.
   - Useful when we want to access the element immediately before the current element.
   - HTML Example: <label>Title</label><input type="text" />
