-> CSRF (Cross-Site Request Forgery) :
        - In some cases a malicious site in the other tab may tricks a logged-in user’s browser into making a request to another site (like AEM)
          without the user realizing it
        - When you make a POST request to an AEM servlet, you need to include a special token called the CSRF token to ensure that the request is
          legitimate and not coming from a malicious source.
        - So the rule AEM follows is "I will only accept POST requests that include a secret token I generated".
        - Without this csrf token, AEM will reject the POST request with a 403 Forbidden error, even if the user is authenticated.

1. Can we use a Java class in HTL
Ans. Directly we cannot use a Java class in HTL, but we can use it through Sling Model.
        - Write the logic in JAVA CLASS --> Use it in sling model by creating OBJ --> Then use it in HTL with the help of model.

--------------------------------------------------
2. Can we use a Service in JAVA Class?
Ans. - we cant use it directly using @Reference
     approach-1: Using Constructor:  Instead we can send the service as a parameter to the JAVA class "constructor" and then we can use it in that class.
              -- Java Service --> Sling model - send the service in obj constructor --> Use the service in JAVA class
              EG : Model : com/aem/geeks/core/models/impl/BhargavModelServiceClassTestingModelImpl.java
                   Service : BhargavModelServiceClassTestingServiceImpl.java
                   Java Class : BhargavServiceInJavaToModel.java

    approach-2: Using Bundle Context : The BundleContext queries the OSGi service registry and returns the registered service.
                -- Service class have the logic --> Java class gets hold of service using Bundle context --> sling model uses the java class using obj
                EG: Service : BhargavModelServiceClassTestingServiceImpl.java
                    Java Class : BhargavServiceInJava.java
                    Model: BhargavModelServiceClassTestingModelImpl.java

--------------------------------------------------
3. Can we use a sling model in java clas ?
Ans. Directly we cant use, Indirectly we can use but it is discouraged.
        - Similar to the above example, here also we send the sling model to java class using Constructor.
            EG :  ProductUtil util = new ProductUtil(this);  //This represents to the current clas object.

--------------------------------------------------
4. Can we use a Service directly in HTL?
Ans. no we cannot use it directly, we need a sling model.

*) ANNOTATIONS:

@Activate : when we want something to load or execute at the time of that particular service initializations, there we use this. for loading init type method.

@PostConstruct : A method with this annotation is called after all the injections are done. It is mainly used for initialization logic.

- immediate=true When we want to activate the service immediately after the bundle gets started in that place we use this.

@Modified : It is a lifecycle callback method annotation used in OSGi components. It marks a method that should be called
            whenever the OSGi configuration for that component is changed at runtime, without restarting the bundle.
- In simple words, When you change configuration values, AEM will automatically re-invoke the method annotated with @Modified, So your service picks up
  the new configuration immediately — without redeploying or restarting AEM.

@ValueMapValue : Specifically used to get the node properties of the resource. for access node props, this is safer and cleaner than @inject.

@Inject : Inject values from AEM resource in to model fields. it is more flexible and injects almost anything like requestAttributes, Resource Props etc.

@Default : provides a default value if the actual property value is missing.

@SlingObject : Injects Sling objects such as ResourceResolver, Resource, SlingHttpServletRequest.

@OSGiService : Injects an OSGi service into the Sling Model.

@ScriptVariable : Used in aem to get built-in objects from the htl script, like currentPage, resource, PageManager

@Component : Marks a class as an OSGi service component. used for registering our service in osgi.

@ObjectClassDefinition(name, definition) : used for defining osgi configuration.

@AttributeDefinition : used for defining fields in the configuration.

@Designate : Links configuration class to the component (used for OSGi configs).

@Configuration : Used for craeting Context-Aware Configuration, which is different from OSGi configuration.

@ChildResource : @ChildResource in Adobe Experience Manager Sling Models is used to inject child resources (nodes) of the current resource directly
                 into your model—instead of you manually traversing the JCR tree.
                Eg : @ChildResource(name = "multifieldName")
                     private List<ChildModel> items;

@ResourcePath(path="/content/aemgeeks/us/en/home") : - Instead of injecting the current resource, Sling fetches the resource located at the given path
                                                     - This simply means : Resource resource = resourceResolver.getResource("/content/we-retail/us/en");

@RequestAttribute(name = "rAttribute") : - It injects an attribute stored in the HTTP request.
                                         - It simply means request.getAttribute("rAttribute")
                                         - it is used to get a value that is passed only for that specific request.

@Self & @Via

Note : First understand the situation
* A Sling Model is created using something called an adaptable.
* @Model(adaptables = SlingHttpServletRequest.class)
        - Sling must sometimes be told where to take data from.

* When Sling creates the model:
  - It uses the adaptable (request or resource) to construct the model
  -(Imp) Then it processes every field separately
  -(Imp) Each field needs its own injection instruction
  - adaptables: means we're saying to sling that this model can be created using a request or resource. Mainly defines how model is created.
  - @Self: using this we say that store that same request or resource inside this variable. injects that adaptable into a field



@Self : It tells Sling to create this model from the current adaptable (Request or Resource). Give me the same object that created the model.
  - adaptables: means we're saying to sling that this model is created using a request or resource. Mainly defines how model is created.

  - @Self: using this we say that store that same request or resource inside this variable. injects that adaptable into a field.
  - IMP : This @Self says: "don't go look at a child resource, don't call an OSGi service, don't read a value map property — just take the current adaptable itself."
          just adapt the same resource/request you're currently sitting on.


- Work of @Self is :
        -> @Self
           private Resource resource;  this means -- resource = adaptable.adaptTo(Resource.class);

        -> @Self
           private SlingHttpServletRequest request;  This means -- request = adaptable.adaptTo(SlingHttpServletRequest.class);

        -> @Self
           private Teaser teaser;  This means -- teaser = adaptable.adaptTo(Teaser.class);


Eg : @Model(adaptables = SlingHttpServletRequest.class)
     public class MyModel {

         @Self
         private SlingHttpServletRequest request;

         public String getPath() {
             return request.getRequestPathInfo().getResourcePath();
         }
     }
 - If @Self is removed: private SlingHttpServletRequest request;   ---> then request will be null, even though the model was created using the request.

-----------------------------------------------------

@Via : Before injecting, go through this object first.
* It is mainly used because most AEM models are adaptable from request, but properties are stored in resource.

Eg : Model created using request, but property is inside resource.
@Inject
private String title;  // this will mostly gives null

@Inject
@Via("resource")       // This fixes the issue and get the props by looking in component resource properties.
private String title;

* Here the flow is
    - Start with request.
    - Go to request.getResource().
    - Then read property title.

*) SERVLETS - GET and POST requests explanation

Simple understanding
* Post : A POST servlet is used when the client sends data to the server.
            - Save data into JCR
            - Update existing content
            - Trigger some backend processing
            - Submit forms
            - Or in some places it simple send the data to the server and that data we use in our code.

* Get : A GET servlet is used when the client requests data from the server. used for mainly Reading data from JCR.

--> GET FLOW
a) Browser issues GET /path?query=abc HTTP/1.1 (headers, cookies).
b) Server/router receives request, Sling/Servlet container resolves resource/servlet.
c) Container calls doGet(req, res)
d) doGet reads req.getParameter(...), session, headers, etc., prepares content and sets response headers (Content-Type, caching),
writes body via res.getWriter() or stream.
e) Server sends HTTP status (200 OK) + body back.
f) Browser renders or caches per response headers.

--> POST FLOW
a) Browser issues POST /path HTTP/1.1 with body (form-encoded, JSON, multipart) + headers.
b) Transports to the server.
c) Container resolves servlet and calls doPost(req, res).
d) doPost reads req.getParameter() (for form), validates input, performs DB/JCR writes.
f) doPost typically returns: HttpStatus codes and then Client receives response and acts accordingly.

*) req.setAttribute() AND req.getAttribute()  Explanation with Example

Q) What is an “attribute” in a request?
Answer: You can store temporary data in the HttpServletRequest object using setAttribute() and later read it back using getAttribute().
--> Think of a HttpServletRequest (or SlingHttpServletRequest) object as a container or bag that travels from the
    browser → AEM → servlet → JSP → other servlets.
--> That bag can carry
a) Parameters (from the user input, like req.getParameter("name"))
b) Headers (like cookies, content-type)
c) And also... attributes that developers can manually add.

--> So attributes are temporary data stored inside the request object — for this single request only.
--> req.setAttribute("key", value);  => Adds a data item to the request.
--> req.getAttribute("key");  => Retrieves that data item later.

--> Example With code snippets: Imagine you’re processing a registration form.
Step 1: You receive a POST request.
String name = req.getParameter("name");
String email = req.getParameter("email");

- Now, before forwarding this to a JSP page or another servlet, you want to store these values inside the request temporarily

Step 2: Set attributes
req.setAttribute("userName", name);
req.setAttribute("userEmail", email);
- Now your req object carries these attributes.

Step 3: Forward the request
RequestDispatcher dispatcher = req.getRequestDispatcher("/thankyou.jsp");
dispatcher.forward(req, res);

- Then inside your JSP (HTL or JSP):
<p>Welcome ${requestScope.userName}! Your email is ${requestScope.userEmail}</p>  --> Output : Welcome Bhargav! Your email is bhargav@example.com

Final understanding: (temporary notepad that travels along with one request) Here with in the request, until the req is there at that time we are setting
these attr(variables) userName, userEmail with our data.

Note: We can forward this attributs from one servlet to another, like we can set these attributs in one servlet and then get these in another servlet
// Forward the request to another servlet
RequestDispatcher dispatcher = req.getRequestDispatcher("/second");
dispatcher.forward(req, res);

*) RUN MODES
- Run Modes in AEM are a way to customize the behavior and configuration of an AEM instance for different environments (like author, publish, dev,
  stage, prod, etc.) using the same codebase.
        PURPOSE of runmodes:
        *Separate configurations for Author and Publish.
        *Maintain different settings for Dev / QA / Prod.
        *Control which components, workflows, or OSGi configurations are active.
- When we want to identify our AEM instance uniquely, either in code(slingSettingService.getRunModes()) or in osgi configs, we set run modes. run modes are two types.
a) Installation Run modes (permanant run modes) : these we will set while installing aem instance. and we cannot change these runmodes,  these are four
                                                  types
                                                1. Author
                                                2. Publisher
                                                3. SampleContent  - Means aem will give you the basic sample pages like WE-RETAIL
                                                4. NoSampleContent - no sample or demo sites (means no we retail or wknd)
   - Author & Publisher, sampleContent & NoSampleContent are mutually exclusive means we cannot use them together.
   - By default if we do not set anything, aem will automaytically create Author with SampleContent.

b) Customized Run Modes : Runmodes we create like Prod, Stg, Qa, Ciqa

=> Creation of Run Modes, we have 5 ways
1) Sling.Properties : we wont prefer this because we won't have permissions. and it is OOTB file so we should not change.
2) system properties (-D) -Dsling.run.modes=publish,prod,us : this we can create while starting our aem instance
3) Filename Detection - aem-publsh-p4503 (most commonly used): we remane a file, and then while installing the aem will understand what run mode it is.
4) -r option -  java-jar file-r publish,prod,us: here while starting the aem we update the start script so then it will create a run mode,
    (2nd and 4th options are almost same)
Note : we can check the available run modes at /system/console/status-slingsettings

Q) How do you configure OSGi differently for Author and Publish?
ans. “By placing configuration files inside run mode–specific folders like /apps/project/config.author and /apps/project/config.publish.
AEM automatically picks the correct configuration based on its active run mode.”

*) Filter.XML - Important things to know about Filter.XML

- Every Module/bundle have its own filter.xml file, filter.xml tells the content package which JCR paths to include when deploying to AEM and
  What content should be installed from this package. if not included then our AEM will simply be ignored the deployment of those paths even
  if the files exist in your project.
- Whatever the path we have added in it will be deployed to the server at the time of deployment.,
- If you added any new file or folder, then you must add that path in the filter.xml file, otherwise that file or folder won't be deployed to the server.

--> Different Types of mode in filter.xml
1. replace (default) : Replace CRX repository content with code package content after deployment. it will Delete missing nodes.
                       eg:  - Crx repo have - /apps/myproject/a
                                              /apps/myproject/b
                            - and in our code we have or (we can also call in packaged content) - /apps/myproject/a
                            - Then after deploymet it will delete "/apps/myproject/b" in AEM instance CRX, because it is missing in our code.

2. merge : - It merges code package content with CRX repository content after deployment. it will NOT delete missing nodes.
            eg:  - Crx repo have - /apps/myproject/a

                 - and in our code we have - /apps/myproject/b
                 - Then after deploymet it will NOT delete "/apps/myproject/a" in AEM instance CRX, it simply creates or updates /apps/myproject/b and keep both nodes.
                    /apps/myproject/a   -> remains
                    /apps/myproject/b   -> created

3. Update : - It updates existing nodes in CRX repository with code package content after deployment. it will NOT delete missing nodes and also won't create new nodes.
            eg:  - Crx repo have - /apps/myproject/a
                                   /apps/myproject/b
                 - and in our code we have - /apps/myproject/c
                 - Then after deploymet it will NOT delete anything, it simply adds /apps/myproject/c in aem crx
                    /apps/myproject/a
                    /apps/myproject/b
                    /apps/myproject/c

4. replace_properties : It replaces only the properties of existing nodes in CRX repository with code package content after deployment.
                        it will NOT delete missing nodes and also won't create new nodes.
            eg:  - Crx repo have - /apps/myproject/a (prop1=value1)
                                   /apps/myproject/b (prop2=value2)
                 - and in our code we have - /apps/myproject/a (prop1=newValue)
                 - Then after deploymet it will NOT delete anything, it simply updates the prop1 value of /apps/myproject/a in aem crx
                    /apps/myproject/a (prop1=newValue)
                    /apps/myproject/b (prop2=value2)

*) adaptTo()

* It is simply like a variable converting thing, eg: Node node = resource.adaptTo(Node.class);  -> here we are converting our resource to node.

* Different Types:
1. Resource → adaptTo(ValueMap.class)
 - Use when: you only need properties of the resource. use when we just want the properties stored on this resource.
 - Eg: Read title, description, custom component fields, Fetch simple configuration values, Access metadata properties quickly.
 - Code eg : ValueMap props = resource.adaptTo(ValueMap.class);
             String title = props.get("jcr:title", String.class);

2. Resource → adaptTo(Node.class)
 - Use when: you need low-level JCR operations.
 - eg: Add/remove nodes manually, Set properties using JCR APIs.
 - Code eg: Node node = resource.adaptTo(Node.class);
            node.setProperty("myProp", "value");

3. Resource → adaptTo(Page.class)
 - Use when: the resource represents a page and you need page-specific methods.
 - eg: Get page title, template, parent page, Navigate site structure, Check if page is valid or published.
 - Code eg: Page page = resource.adaptTo(Page.class);
            String title = page.getTitle();

4. Resource → adaptTo(Asset.class)
 - Use when: the resource represents a DAM asset and you need asset functionality.
 - eg: Read asset metadata, Get renditions (original, thumbnails), Access binary streams, Add new renditions.
 - Code eg: Asset asset = resource.adaptTo(Asset.class);
            String mimeType = asset.getMimeType();

*) Understanding -- "GSON"
- Gson is a Java library developed by Google that converts: "Java Object → JSON" called Serialization and "JSON → Java Object" Deserialization.
- Why do we need Gson? : Because Java and JSON are different formats and java cant understand json response directly,
                         and Gson helps us bridge that gap. It allows us to easily convert complex Java objects into JSON strings that
                         can be sent over the network or stored, and also to convert JSON strings back into Java objects for processing.
- Very Important: Field Names Must Match between JSON and Java Class.

- Using GSON
    -> Create a java class
        public class TodoItem {
                                    // the below three fields are the properties of the json response we are getting from the API,
                                    so we have to create the same fields in our java class
            private int id;
            private String todo;
            private boolean completed;

            // getters setters
        }

    -> Then we give
        Gson gson = new Gson();
        TodoItem item = gson.fromJson(response, TodoItem.class);

    -> How Gson Knows Where to Put Data
        - Eg json : {
                        "id": 1,
                        "todo": "Buy groceries",
                      }
        - When Gson see's - gson.fromJson(response, TodoItem.class);, then it goes to that TodoItem class and looks for the fields,
          and then it matches the json properties with the java class fields, and then it put the values in those fields.
        - It compares json property "id" with java class field "private int id;" and then assigns its value, and same for other properties.
        - This complete process is called de-serialization, means converting json response to java object.

EXAMPLE : (IN DEPTH EXPLANATION)
-> API RESPONSE : {
                    "todos": [
                      {
                        "id": 1,
                        "todo": "Learn Java"
                      },
                      {
                        "id": 2,
                        "todo": "Learn Gson"
                      }
                    ]
                  }

- Java Class for above json response : public class TodoResponse {

                                           private List<TodoItem> todos;

                                           public List<TodoItem> getTodos() {
                                               return todos;
                                           }
                                       }

- Java Class for TodoItem : public class TodoItem {
                                    private int id;
                                    private String todo;

                                    // getters and setters
                                }

- Now when this line executes : TodoResponse responseObj = gson.fromJson(jsonString, TodoResponse.class);

     - Gson internall does this :
          1. Creates response object of type TodoResponse. - TodoResponse responseObj = new TodoResponse();
          2. Then it goes to json key "todos" and then finds the matching field in our java class which is "private List<TodoItem> todos;".
          3. then it creates a list internally - List<TodoItem> todoList = new ArrayList<>();
          4. then reads json object {"id": 1, "todo": "Learn Java"} and then creates a TodoItem object - TodoItem item1 = new TodoItem();
          5 then sets the values in that object - item1.setId(1); item1.setTodo("Learn Java");
          6. then adds that item to the list - todoList.add(item1); similarly it does for all the json objects in the array.
          7. now it assigns that list to the response object - responseObj.todos = todoList;
          8. now our "responseObj" contains -- TodoResponse
                                                |
                                                --> todos
                                                      |
                                                      --> TodoItem(id=1,todo="Learn Java")
                                                      --> TodoItem(id=2,todo="Learn Gson")
    - so now when you call: responseObj.getTodos() it returns List<TodoItem> which contains : [
                                                                                               TodoItem(id=1,todo="Learn Java"),
                                                                                               TodoItem(id=2,todo="Learn Gson")
                                                                                              ]


----------------------------------------
- Serialization : converting Java Object → JSON
        - Suppose we have a java object of TodoItem class, and we want to convert it to json string, then we will do like this :
            TodoItem item = new TodoItem();
            item.setId(1);
            item.setTodo("Buy groceries");
            item.setCompleted(false);

            Gson gson = new Gson();
            String jsonString = gson.toJson(item);
        - Here gson.toJson(item) will convert the java object "item" to json string like this :
            {
                "id": 1,
                "todo": "Buy groceries",
                "completed": false
            }

GRAPHQL

- In AEM, GraphQL is mainly used when your content is stored as Content Fragments and some external application needs that content.
- Usually GraphQL queries are written in the external application, not in AEM.
        -> The React or other external applications sends this query to AEM using post request. and aem takes that query reads it and sends response.

- The work of traditional api's and graphql is almost same, sending the content to the external application, then whats the advantage of GRAPHQL
        -> lets say If a React app wants only the title and author, traditional APIs may still send complete data (we can write the code to filter
           out the ones that are not reqd but that is not a advaisable standard)
                json : {
                         "title": "AEM Basics",
                         "author": "Bhargav",
                         "description": "Learning AEM GraphQL"
                       }
        -> But with graphql,
            - The React app can ask:
                Graphql: {
                           articleByPath(
                             _path: "/content/dam/articles/aem-basics"
                           ) {
                             title
                             author
                           }
                         }

            - AEM returns only:
                    {
                      "title": "AEM Basics",
                      "author": "Bhargav"
                    }
            - The consumer can decide what fields it wants.

- -> GRAPHIQL EDITOR : http://localhost:4502/aem/graphiql.html
  -> To see graphql endpoints in crxde : /content/cq:graphql/global/endpoint
  -> to create graphql endpoint : http://localhost:4502/libs/cq/graphql/sites/admin/content/console.html

- Types the external application request the data:
        1. Persisted Query
                - we write a query in our graphql editor and save that file and aem create a url for that file containing query,
                  that url is called persisted query url.
                - React or external application simply uses that url and requests the data, without writing the whole query.
                - eg: we created : http://localhost:4502/graphql/execute.json/global/bhargavFirstQuery
                      in react we simply call : GET /graphql/execute.json/global/bhargavFirstQuery
                - This is advisable because it is simple and faster.

        2. In react we write the entire query and sends it in POST request. this is not advisable because in every request we send the query
           and it make the system slower.

- You can find the example graphql queries in com/aem/geeks/core/Bhargav-notes.txt

*) DISPATCHER   -- C:\Documents out side onedrive focus\AEM RELATED\aemgeeks\dispatcher.any

--> Important points
- Dispatcher does not perform tasks, dispatcher is a module in our apache web server that tells Apache webserver what tasks need to perform.
- Caching, load balancing, access and security all these are the feature of APACHE webserver, but the dispatcher tell the server
  to create the cache as per the provided configurations.
- Using rewrite rules we will redirect the pages :: Eg: if we hit www.intel.com  -> it will redirects to -> https://www.intel.com/content/www/us/en/homepage.html

---> Difference between cache deletion, cache invalidation, cache purge
- Cache Deletion : Physically removing cached files from filesystem. so the next request goes to publisher and then page served and a new cache file got created.

- Cache Invalidation : Marking cache as stale or unusable without deleting file immediately.
                        - In this dipatcher checks the ".stat timestamp > cached file timestamp", then cached file considered outdated
                        - dispatcher fetches fresh content and overwrites cache
                        - Old file exists physically but is logically invalid.

- Cache Purge : “Cache purge” is most commonly associated with CDNs.
                 - it usually means Explicit forced removal of cached content from caching layer like cdn.

- cache flush : It basically means cache invalidation happened.

          │                 ├──► getProperty("jcr:title") → read
          │                 ├──► setProperty("prop", "value") → write
          │                 └──► getSession() → obtain Session
          │
          └──► adaptTo(Session)
                     │
                     ▼
              [ JCR Session ]
                     │
                     ├──► getNode("/path/to/node")
                     ├──► save()   ← commits changes
                     └──► logout() ← closes session

--> Common Flows

-> READING
        ResourceResolver → getResource(path)
               ↓
        Resource → adaptTo(Node)
               ↓
        Node → getProperty("propName")

-> WRITING
        ResourceResolver → getResource(path)
               ↓
        Resource → adaptTo(Node)
               ↓
        Node → setProperty("prop", "value")
               ↓
        Node.getSession().save()


-> Just One Line Summary
        ResourceResolverFactory
           ↓
        ResourceResolver
           ↓ getResource()
        Resource
           ↓ adaptTo(Node)
        Node
           ↓ getSession()
        Session.save()
